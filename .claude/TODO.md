- **2026-05-10** — `JwtAuthenticationFilter` does NOT re-validate the account status (`enabled`/`locked`) on every request — it trusts the signed JWT. To re-wire if hardening needed.
- **2026-05-10** — `UserDetailsServiceImpl` returns a Spring `User` (no custom wrapper around `Account`). Consequence: the login service must redo a `findByUsername` to build the JWT `UserPrincipal`.
- **2026-05-10** — `UserPrincipal.email` renamed to `username` to stay consistent with `Account.username`.
- **2026-05-11** — No dedicated `Administrator` entity: a SaaS admin = root `Utilisateur` (without `Proprietaire`/`Employe` subclass) + `Account` with an "ADMIN" `Role`. Runtime distinction via `Account.role`. If future admin-specific fields are needed → create `Administrator extends Utilisateur` then.
- **2026-05-11** — **Pragmatic hexagonal refactoring**: previous decision (2026-05-09) on `JpaRepository` in `domain/repository/` **reverted**. From now on: port `<Entity>Repository extends BaseRepository<E>` in `domain/repository/` (pure interface), adapter `<Entity>JpaRepository extends JpaRepository, <Entity>Repository` in `infrastructure/repository/`. `GlobalService` typed on `<Entity>Repository`. JPA entities stay in `domain/model/` (assumed compromise). Typo `repositorty/` fixed everywhere.
- **2026-05-11** — **FR/EN backend i18n** in place: default `Locale.FRENCH`, locale resolved via `Accept-Language` header (`AcceptHeaderLocaleResolver`). All custom exceptions (except `AuthentificationException` which inherits Spring Security) now extend `LocalizedRuntimeException` and carry `(messageKey, args)`. The `GlobalException` advice resolves via `IMessageSourceService`. Jakarta Bean Validation keys overridden in `messages*.properties`. `useCodeAsDefaultMessage=true` → if a key is missing, the code is returned as is (safe fallback). Convention: prefix keys by domain (`entity.*`, `validation.*`, `error.*`, `auth.*`, etc.).
- **2026-05-11** — **Mandatory rules established** (all saved in user memory):
  - Controller: `public static final String BASE_PATH = "/api/v1/<scope>"` (hardcoded version) + `@RequestMapping(BASE_PATH)`, no `@PathVariable("version")` on the handlers
  - Every application service has a **mandatory** interface `I<X>Service` + impl `<X>ServiceImpl`; injection sites use the interface
  - DTOs: suffix `<X>Request` (input) or `<X>Response` (output), never `Info` nor generic `Dto`
  - Method parameter of DTO type: name = full camelCase of the type (`AccountRequest accountRequest`), not generic `info`/`request`/`dto`
  - Tests: every application service has an `<X>ImplTest` (JUnit5 + Mockito), every controller a `<X>ControllerTest` (`@WebMvcTest` or `MockMvcBuilders.standaloneSetup`)
- **2026-05-11** — **`entreprise` module**: `Entreprise` sub-domain extracted from the `magasin` module for business cohesion (the company is a root multi-tenant aggregate, distinct from the store). The `magasin` module now only carries `Magasin`. Cross-references added (`Magasin.entreprise`, `Entreprise.magasins`).
- **2026-05-11** — **Refresh token**: opaque UUID persisted (`refresh_token` table, duration `properties.expiration().refreshToken()` = 7d). **No rotation** for MVP. `/auth/logout` revokes (idempotent). The JWT access token cannot be revoked server-side (stateless) — it expires on its own in 1h.
- **2026-05-11** — **Spring Boot 4 — restructured test packages**: `@WebMvcTest` is no longer in `spring-boot-starter-test`, you need `spring-boot-starter-webmvc-test`. `@MockBean` removed → replaced by `@MockitoBean` (`org.springframework.test.context.bean.override.mockito`). Current solution: `MockMvcBuilders.standaloneSetup()` for controller tests (no additional pom dependency).
- **2026-05-12** — **Flyway / Spring Boot 4**: Flyway autoconfig is out of `spring-boot-autoconfigure`. You need `spring-boot-starter-flyway` (dedicated SB 4 module) + `flyway-database-postgresql`. `baseline-version: 1` so that existing DBs (already populated by `ddl-auto: update`) consider V1 as already applied. `V1__init_schema.sql` hand-written (40 tables, JOINED `person`/`utilisateur`, `role_permission`, FK via `ALTER TABLE` at the end).
- **2026-05-12** — **Strengthened service isolation**: all `<X>ServiceImpl` inject their `<X>DomainService`, never an `<X>Repository`. DomainServices expose custom queries (`findByUsername`, `findByToken`, `findByLibelle`, `findFirstTrialActif`, `findAllByRoleId`) — return `Optional<E>`, the app service throws. `UserDetailsServiceImpl` (Spring Security) follows the same rule.
- **2026-05-12** — **Custom validators** (`org.store.common.validation`): `@Phone` (SN regex `^(70|75|76|77|78|33)\d{7,9}$`), `@EnumValue(enumClass, ignoreCase)`, `@DatePattern(pattern)` (SMART mode), `@Uuid`. All skip null/empty (combine with `@NotBlank` if required). Each validator has its i18n key + parameterized test.
- **2026-05-12** — **`PermissionCode` enum**: centralized permission codes (`PROPRIETAIRE_ACCESS`, `EMPLOYE_ACCESS`, `EMPLOYE_CREATE`). The `code` in DB = `enum.name()`. Overload `UserPrincipal.hasPermission(PermissionCode)`. No more scattered permission strings.
- **2026-05-12** — **`UserPrincipal.role`**: String field added (role label, not UUID). Serialized in the JWT via `Claim.ROLE("role")`. `UserPrincipalFactoryImpl` uses `account.role.libelle`. (Decision: label > UUID for the flexibility of adding roles.)
- **2026-05-12** — **`ICurrentUserService`**: replaces `@AuthenticationPrincipal UserPrincipal` at the controller. Business services call `currentUserService.getCurrent()` → testable with a simple mock (vs SecurityContext setup + custom argument resolver in MockMvc).
- **2026-05-12** — **Data-driven employee rules**: no `"MANAGER"` label hardcoded in `EmployeServiceImpl`. The "elevated role = can create employees" rule is based on the `EMPLOYE_CREATE` permission. Adding a future `CAISSIER` (with `EMPLOYE_ACCESS`) or `SUPER_MANAGER` (with `EMPLOYE_CREATE`) requires **no application change** — only an `ensureRole(...)` in `DataInitializer`.
- **2026-05-12** — **`<X>Response(<X> entity)`**: every output DTO must expose a secondary constructor that extracts the fields from the entity. Centralizes mapping, removes duplication in services.
- **2026-05-12** — **`SELECT new <X>Response(entity)` in `@Query`**: JPQL projection via the Response's secondary constructor, never the field list spelled out (DRY, no desync on field addition). Prerequisite: the `Response(Entity)` convention must be applied.
- **2026-05-12** — **`create()` in the DomainService**: entity construction (`new` + setters + `save`) lives in `<X>DomainService.create(...)`. The app service delegates + applies the business rules. Cleanly separates "persistence" (domain) from "orchestration" (application).
- **2026-05-12** — **`impl` package reorganization**: all `<X>ServiceImpl` classes moved to `<module>/application/service/impl/`. Cross-cutting consistency (interface in `service/`, impl in `service/impl/`).
- **2026-05-12** — **No private methods in application services**: any factoring becomes a public method of `I<X>Service`, parameterized rather than specialized (e.g., `createAccount(request, String roleName)` vs `doCreate(request)` which hardcoded `ROLE_PROPRIETAIRE`). Tolerated exceptions: trivial helpers on DTO/utility, not a business service.
- **2026-05-12** — **No business logic in the controller**: handler = `return service.method(request)`. No cast, no `new Response(...)`, no getter-chain, no multi-service orchestration. If the necessary business method creates a DI cycle, create it in the service that already orchestrates (seen: `IRegisterPropertyService.registerEntrepriseByAdmin` rather than `IEntrepriseService.createByAdmin`).
- **2026-05-12** — **Response sub-DTO for sub-entity**: when an `<X>Response` aggregates ≥ 3 fields of another reusable entity, create a dedicated `<Y>Response` (e.g., `AccountResponse.user: UserResponse`) rather than putting the fields flat. Hierarchical JSON, DRY, follows the `Response(Entity)` convention.
- **2026-05-12** — **Strategy pattern for sub-type dispatch**: avoid `if (x instanceof A) ... else if (x instanceof B)` in dispatch code. Pattern: interface `<X>Strategy { Class<? extends Parent> targetType(); <Y> resolve(Parent x); }` + `@Component` impls + Spring injects `List<Strategy>` + dispatch via `.filter(s -> s.targetType().isInstance(x)).reduce((a,b) -> a.targetType().isAssignableFrom(b.targetType()) ? b : a)` → "most-specific wins". Allows fallback on the parent type.
- **2026-05-12** — **Additive ERP seed**: the 4 legacy permissions (PROPRIETAIRE_ACCESS, EMPLOYE_ACCESS, EMPLOYE_CREATE, ADMIN_ACCESS) kept in DB so the current Java code keeps working. The 75 new ones (MODULE_ACTION) coexist. Progressive migration of `@PreAuthorize` to do later.
- **2026-05-12** — **`Proprietaire` has no `magasinId`** in `UserPrincipal`: an OWNER owns all their company's stores, not a specific one. `magasinId` is only carried by `Employe` (Manager/Vendeur).
- **2026-05-12** — **DomainService = data access port**: every CRUD or custom query is in `<X>DomainService`. The application never sees `<X>Repository`. A reusable method (e.g., `extractPermissionCodes`) must live in the service that owns the aggregate, never as a private helper duplicated across services.
